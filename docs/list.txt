notas importante auth:
Nota: Hay que manejar user_dict.password_hash pero el usuario retornado por el repo es un objeto ORM que tiene password_hash. Ajustaremos en el repositorio concreto para que el método get_by_email devuelva la entidad con el hash. Mejor modificar el servicio para que reciba el hash del usuario ORM. En el código final se usará el repositorio que devuelve el modelo ORM y adaptaremos.

Optemos por implementar el auth_service con los repositorios concretos y usando el ORM directamente para evitar doble mapeo. Dado que estamos en una implementación limpia pero pragmática, podemos tener una capa de adaptación. Sin embargo, para mantener la pureza, el servicio de aplicación puede esperar entidades de dominio y el repositorio abstracto devuelve entidades de dominio. Pero en el repositorio concreto SQLAlchemy, mapearemos el modelo a la entidad. Para simplificar, en este tutorial haremos que el repositorio concreto devuelva directamente el modelo ORM (que es similar a la entidad) y lo usaremos en el servicio. Dado que la lógica de negocio central está en el estado del servicio (transiciones de estado), que no requiere mucha inteligencia, podemos hacer una concesión práctica.

Voy a reescribir el auth_service de manera directa usando los repositorios que manejan los objetos ORM (que tienen atributos como password_hash). Así mantendremos la coherencia con el resto.
